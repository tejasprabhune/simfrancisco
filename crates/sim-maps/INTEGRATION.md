# Golden Future Map — Frontend Integration Guide

This document describes every asset, database schema, coordinate system, rendering
rule, and the character-sprite layout the frontend needs to display the San Francisco
tilemap and the entities walking on it. It is the source of truth; where it disagrees
with older notes, trust this file (it matches the `src/render.rs` reference renderer).

The art was re-themed to the **LimeZu "Modern Exteriors" 32×32** set. The ground, the
buildings, and the trees all come from that one cohesive palette. Character sprites use
the **16×16 RPG character** pack. See **§15 Credits / licensing**.

---

## 0. What changed vs the previous art (changelog)

If you integrated against the earlier version, here is what moved:

- **`atlas.png`** — same dimensions (1504×480, 47 cols × 15 rows, 32px) and same
  sampling, but the **tile content is now MX art** and the **47 variants per class are
  now distinct** blob-47 autotile tiles (organic rounded edges), not 47 identical tiles.
- **`facade_atlas.png`** — **changed from 160×96 to 192×128** and the layout is
  different (3 wall colours × 6 slots + a roof row). See §10.
- **`props_atlas.png`** — **new** (288×128, three trees). See §11.
- **Building rendering is new** — per-building roof colours (graph-coloured so
  neighbours differ), composed MX facades that scale with tier, a 3-D bevel, a drop
  shadow, north-first painter order, and an optional facade shear on diagonal streets.
  See §9.
- **`tiles.db` is unchanged** — same schema, same bytes. Only art + rendering changed.

## 2. Coordinate systems

### UTM 10N (metres) — EPSG:32610
All geographic data is projected to UTM Zone 10N (linear metres over SF).

| Parameter | Value |
|-----------|-------|
| Bounding box min | (541 826, 4 172 448) |
| Bounding box max | (558 483, 4 187 294) |
| Width × Height | 16 657 m × 14 846 m |

### Chunk grid
67 × 60 = **4 020 chunks**, each 250 m × 250 m.

```
chunk (cx, cy) → UTM SW corner:
  x = 541826 + cx * 250
  y = 4172448 + cy * 250
```
`cx = 0` is westernmost; `cy = 0` is **southernmost** (UTM y increases north).

### Cell coordinates (within a chunk, LOD 0)
Each chunk is 125 × 125 cells at 2 m/cell.

```
cell (col, row) in chunk → UTM:
  x = chunk_origin_x + (col + 0.5) * 2.0
  y = chunk_origin_y + (125 - row - 0.5) * 2.0    // row 0 = NORTH edge
```
`row 0` is the **northernmost** row (matches image y-down convention).

### lat/lon → chunk (for placing the camera / an entity from GPS)
Project WGS-84 (lon, lat) → UTM 10N (EPSG:32610), then:
```
cx = floor((utm_x - 541826) / 250)
cy = floor((utm_y - 4172448) / 250)
```
Example iconic chunks: Alamo Square (32, 34), FiDi/Transamerica (43, 43),
Ferry Building (46, 43), SoMa/Market St (43, 35).

### Image pixel coordinates
When a chunk is rendered at `S` px/cell (the reference detail renderer uses S = 32):
```
within-chunk pixel of cell (col, row):  px = col * S,  py = row * S
```
A full-city image places chunk `(cx, cy)` at `(cx * 125 * S, (ny-1-cy) * 125 * S)`
(the `ny-1-cy` flips south-up chunk rows to north-up image rows).

## 3. LOD levels

| LOD | Factor | Cells/chunk | m/cell |
|-----|--------|-------------|--------|
| 0 | 1× | 125 × 125 | 2 m |
| 1 | 2× | 63 × 63 | ~4 m |
| 2 | 4× | 32 × 32 | ~8 m |
| 3 | 8× | 16 × 16 | ~16 m |
| 4 | 16× | 8 × 8 | ~32 m |
| 5 | 32× | 4 × 4 | ~64 m |

The `buildings` table stores records at **LOD 0** only; scale footprint coords by
`1/factor` for other LODs. (Buildings are typically only drawn at LOD 0–1.)

## 4. tiles.db schema (unchanged)

```sql
CREATE TABLE chunks (
    cx INTEGER, cy INTEGER, lod INTEGER,
    render    BLOB NOT NULL,   -- zstd( little-endian u32[w*h] ), row-major, row 0 = north
    collision BLOB NOT NULL,   -- zstd( u8[w*h] )
    w INTEGER, h INTEGER,
    PRIMARY KEY (cx, cy, lod)
);
CREATE TABLE meta (key TEXT PRIMARY KEY, value TEXT NOT NULL);
CREATE TABLE buildings (
    cx INTEGER, cy INTEGER,
    cell_x REAL, cell_y REAL,   -- SW corner of footprint, chunk-local cells (cell_y from SOUTH)
    cell_w REAL, cell_h REAL,   -- footprint size in cells
    tier INTEGER,               -- 0 Low, 1 Mid, 2 Tall
    PRIMARY KEY (cx, cy, cell_x, cell_y)
);
```

### Decoding a `render` BLOB
`zstd`-decompress → little-endian `u32[w*h]`, row-major, row 0 = north. Each `u32`:
```
class   = tile_id & 0xFF          // SemanticClass 0..14
variant = (tile_id >> 8) & 0xFF   // blob-47 autotile variant 0..46
// bits 16..31 reserved (zero)
```

### Decoding a `collision` BLOB
`zstd` → `u8[w*h]`. 0 free · 1 park/shoreline · 2 grass · 8 stairs · 255 BLOCKED
(water/cliff/building). Intermediate values come from slope cost modifiers.

## 6. Main atlas (`assets/atlas.png`)

**1504 × 480 px = 47 columns × 15 rows, each tile 32 × 32 px.** Sampling is unchanged:

```
atlas_x = variant * 32      // 0..46
atlas_y = class   * 32      // 0..14
```

Each row is one semantic class, sourced from an MX tile (row = class ordinal from §5:
0 Grass, 1 ParkGrass=grass tinted darker, 2 Sand, 3 Path/dirt, 4 Sidewalk, 5 Road/asphalt,
6 Stairs, 7 CliffFace, 8/13/14 building roofs, 9 BuildingWall, 10 Water, 11 Plaza, 12
Shoreline). The 47 columns are the **blob-47 autotile variants** (organic rounded edges
with a subtle palette rim), so class boundaries read as defined contours instead of a
flat checkerboard.

The only contract the frontend needs is the `(class, variant) → (x, y)` formula above
(full source mapping is in `build_atlas.rs`). `variant = 46` is the fully-surrounded
interior fill — use it as the representative solid colour of a class.

---

## 7. Autotile variant system (blob-47)

The pipeline pre-computes a blob-47 variant per cell. Bit layout of the raw 8-neighbour
mask (a bit is set when that neighbour shares the cell's class):

```
bit0 = N   bit1 = NE   bit2 = E   bit3 = SE
bit4 = S   bit5 = SW   bit6 = W   bit7 = NW
```
Corner bits are suppressed unless **both** flanking cardinals also match, leaving the 47
valid configurations → variant indices 0..46. The frontend does not need to recompute
this — the variant is already baked into each tile_id. (It only matters if you author
new tiles: each variant tile's open/rounded edges correspond to its mask.)

---

## 8. Render order (the whole frame, top level)

Draw in this order. `S` = your chosen pixels-per-cell (the reference renderer uses 32).

1. **Ground** — one atlas tile per cell (§6).
2. **Building drop shadows** — all buildings in the visible chunks (§9.3).
3. **Buildings** — facades then roof fills, **north-first** (§9.1–9.5).
4. **Roof bevel edges** — all buildings, on top (§9.6).
5. **Trees** — scattered on grass/park cells, north→south (§11).
6. **Entities / characters** — depth-sorted by world-y (§13), feet anchored to the cell.

The reference implementation of steps 1–5 is `render::render_detail_chunk` in
`src/render.rs`; the formulas below are everything you need to reproduce it.

## 9. Building rendering (the big change)

Each row of the `buildings` table is **one discrete building**. Query per chunk,
**north-first** (so nearer/south buildings paint over farther/north ones):

```sql
SELECT cell_x, cell_y, cell_w, cell_h, tier FROM buildings
WHERE cx = ? AND cy = ? ORDER BY cell_y DESC;
```

### 9.1 Footprint and facade rectangles (at 32 px/cell; scale by S/32 for other zooms)
`h` = chunk height in cells (125 at LOD 0).
```
img_top_row   = max(0, floor(h - cell_y - cell_h))      // cells from north
img_south_row = clamp(floor(h - cell_y), 0, h)          // south edge, cells from north
fx = floor(cell_x * 32)
fw = max(1, floor(cell_w * 32))
footprint_rect = (fx, img_top_row*32, fw, (img_south_row - img_top_row)*32)   // the roof
facade_top_y   = img_south_row * 32                     // facade hangs BELOW here
```

### 9.2 Tier → facade height (floors)
```
facade_height_px(tier) = floors(tier) * 32,  floors = { Low(0): 2, Mid(1): 4, Tall(2): 7 }
```

### 9.3 Drop shadow (sun top-left) — drawn first, on the ground
For the silhouette rect `(fx, footprint_top, fw, fh + facade_height_px(tier))`, with
`s = 5` px, darken existing ground pixels by ×0.55 in an L offset down-right:
- right band:  `x ∈ [fx+fw, fx+fw+s)`, `y ∈ [top+s, top+fh+facade_h+s)`
- bottom band: `y ∈ [bottom, bottom+s)`, `x ∈ [fx+s, fx+fw+s)`

### 9.4 Per-building colours
A deterministic hash picks the **wall** colour and the *preferred* **roof** colour:
```
FNV-1a over [cx, cy, round(cell_x*4), round(cell_y*4)]  (64-bit, seed 0xcbf29ce484222325, prime 0x100000001b3)
wall_variant      = (hash >> 7)  % 3      // 0 brick, 1 blue, 2 taupe   (rows of facade atlas)
roof_pref         = (hash >> 23) % 4      // 0 terracotta,1 slate,2 olive,3 grayblue
```
The **final roof colour is graph-coloured** so no two touching buildings share one:
process the chunk's buildings in the query order; two footprints are adjacent if their
rects (padded by 1 px) overlap; assign each building its `roof_pref` unless an
already-assigned neighbour uses it, else the lowest free colour 0..3. (Planar adjacency
is always 4-colourable, so this never runs out.) This is what makes abutting buildings
read as separate structures.

### 9.5 Facade composition (`draw_facade`)
Draw the facade below `facade_top_y`, `fw` wide, `facade_height_px` tall, in 32×32
tile-columns. Slot atlas coords come from §10. For a column of `n_rows` tiles:
```
row 0           → CORNICE   (only if n_rows >= 3; else the top row is a window)
row n_rows-1    → BASE
otherwise       → window:  p = (n_rows>=3 ? row-1 : row);  p even → WIN_TOP else WIN_BOT
sample facade_atlas at ( slot*32 + dx , wall_variant*32 + dy ),  skip if alpha <= 64
```
**Optional shear on diagonal streets (§9.7).** If you skip the shear, every column uses
`n_rows = facade_height_px/32`.

### 9.6 Roof fill + bevel
- **Fill**: tile the roof colour across the footprint — sample `facade_atlas` at
  `( roof_color*32 + (dx mod 32), 3*32 + (dy mod 32) )` (roof row = 3).
- **Bevel** (separate pass, after *all* roof fills so a neighbour can't cover it):
  within the footprint, darken the **right/bottom 3 px** by ×0.5 (self-shadow + seam)
  and lighten the **top/left 2 px** by ×1.14 (sun highlight).

### 9.7 Facade shear (follow diagonal roads) — optional but recommended
To make a building that fronts a diagonal street lean with it: estimate the road slope
under the south edge, then make downhill tile-columns taller so the *base* follows the
road while the cornice stays flush with the roof.
```
slope: for each footprint column, scan up to 10 cells south for the first Road (class 5)
       cell; record its depth. Least-squares slope of depth-vs-column, clamped to ±0.5,
       0 if fewer than 3 columns hit a road.
per tile-column tc: ext = clamp(round(slope*32*tc) - min_over_columns, 0, 64) px,
                    n_rows = (facade_height_px + ext) / 32
```

## 10. Facade atlas (`assets/facade_atlas.png`)

**192 × 128 px = 6 cols × 4 rows, each tile 32 × 32 px.**

- **Rows 0,1,2 = wall colour schemes**: 0 brick-red, 1 blue-gray, 2 taupe.
  - col 0 `CORNICE` — top trim where wall meets roof
  - col 1 `WIN_TOP` — upper half of a window
  - col 2 `WIN_BOT` — lower half of a window
  - col 3 `BASE` — ground-floor / foundation band
  - col 4 `WALL` — plain wall (spare; col 5 duplicate)
- **Row 3 = roof fill colours** (cols 0..3): 0 terracotta, 1 slate-blue, 2 olive, 3 gray-blue.

```
facade slot:  src = ( slot_col*32 , wall_variant*32 )   // slot_col ∈ {0..3}, wall ∈ {0,1,2}
roof colour:  src = ( roof_color*32 , 3*32 )            // roof_color ∈ {0..3}
```
Wall/roof tiles are gently desaturated/lightened so colours read soft, not harsh.

## 11. Props atlas (`assets/props_atlas.png`) — trees

**288 × 128 px = 3 cells of 96 × 128 px** (variants 0,1,2). Each tree is bottom-aligned
and horizontally centred in its cell, with a baked soft shadow and transparent margins.

Scatter trees on **Grass (0)** and **ParkGrass (1)** cells, drawn **north→south** so
southern trees overlap northern ones:
```
hash = FNV-1a over [cx, cy, col, row, 0x7233]   (64-bit)
if (hash % 100) < 7:                            // ~7% density
    variant = (hash / 100) % 3
    src     = (variant*96, 0, 96, 128)
    dst_x   = col*32 + 16 - 48     // centre the 96-wide sprite on the cell
    dst_y   = (row+1)*32 - 128     // trunk bottom at the cell's south edge
    blit with alpha (skip alpha <= 64)
```

## 13. Character sprites (`~/Downloads/16x16-RPG-characters/`)

Player and NPC sprites come from the **16×16 RPG character** pack. Bundle the sheets you
use with the frontend. All facts below are pixel-verified.

### 13.1 Universal rules (walk sheets)
- **Cell size: exactly 16 × 16 px**, no gutters; the character fills the cell
  (≈13 px wide, 16 px tall, bottom-aligned). Background is transparent → blit whole cells.
- A **character block = 48 × 64 px = 3 cols × 4 rows**:
  - **rows = facing**: row 0 DOWN, row 1 LEFT, row 2 RIGHT, row 3 UP.
  - **cols = frames**: col 0 step-A, col 1 **IDLE**, col 2 step-B.
- LEFT and RIGHT are both present; their idles are exact mirrors but the walk steps
  differ — **use the real RIGHT row, don't flip LEFT** for walking.

```
frame (x, y, w, h) within a character block at block origin (bx, by):
  x = bx + frame*16,  y = by + dir*16,  w = 16, h = 16
  dir ∈ {down:0, left:1, right:2, up:3},  frame ∈ {0,1,2}
```

### 13.2 Sheets and how to index a character

| Sheet | Size | Chars | Block grid | Per-char origin |
|-------|------|-------|------------|-----------------|
| `old-style/01-generic.png` | 240×128 | 10 | 5 × 2 blocks (48×64) | `bx = (c%5)*48, by = (c/5)*64` |
| `old-style/02-bard.png … 06-conjurer.png` | 192×128 | 8 each | 4 × 2 blocks | `bx = (c%4)*48, by = (c/4)*64` |
| `old-style/00-guide.png` | 96×64 | 1 (reference) | 6 × 4 cells | use cols 0–2 only |
| `old-style/00-avatars.png` | 384×192 | 8 | 4 × 2 cells of **96×96** | menu/portrait busts; content ≈40×64, anchor bottom-centre |
| `new-style/chibi-layered.png` | 144×48 | 3 (showcase) | 9 × 3 of 16×16 | preview strip, not a clean walk block |

`01-generic` and the `02..06` class sheets are the playable directional sheets. Use
`00-avatars` for character-select / dialogue portraits. Treat `chibi-layered` and
`preview.png` as art previews, not gameplay sheets.

### 13.3 Recommended animation
- **Idle**: show frame 1 (the middle/standing pose).
- **Walk**: cycle `1 → 0 → 1 → 2` at **~8 fps** (≈125 ms/frame) — classic
  contact-pass-contact-pass. Pick the row from the entity's facing.

### 13.4 Placing a character on the map
An entity has a world position (chunk + sub-cell, or a continuous UTM/world coord).
Convert to a screen pixel like any cell (§2), then:
- The map renders at `S` px/cell (e.g. 32). A 16×16 sprite is **half a tile**; scale it
  to taste (e.g. ×2 → 32 px so a character ≈ one 2 m cell, or ×1.5 for smaller chibi).
- **Anchor the sprite's feet** (bottom-centre of the 16×16 cell) to the entity's cell
  centre-bottom, so it stands on the tile.
- **Depth-sort** entities (and ideally interleave with buildings/trees) by world-y:
  draw smaller-y (north) first, larger-y (south) last, so a character south of a tree or
  building is drawn in front of it.
- Use the `collision` layer (§4) to block movement (value 255 = impassable).

## 14. Quick frontend checklist

1. Load `atlas.png`, `facade_atlas.png`, `props_atlas.png`, and the character sheets.
2. For each visible chunk, decode the `render` BLOB → per-cell `(class, variant)`.
3. Draw ground (§6) → shadows (§9.3) → buildings north-first (§9) → bevels (§9.6) →
   trees (§11) → entities (§13), per §8.
4. Query `buildings` per chunk for the building pass; run the greedy roof colouring
   (§9.4) once per chunk (it's deterministic, so it can be cached per chunk).
5. Movement/collision from the `collision` BLOB.

A reference renderer for steps 1–5 (sans entities) is `src/render.rs`
(`render_detail_chunk`); `cargo run --release --bin export_map -- --full` stitches the
whole city to `sf_city.png` for sanity-checking.
